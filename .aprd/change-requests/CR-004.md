# CR-004 — brownfield-adopt: bootstrap ADP foundation on non-ADP projects

> Operator-directed (2026-06-12), approved at self-host loop STEP-5 gate (all Q1–Q3 resolved). Change request against the **existing deliverable** (adds capability; no frozen-artifact version bump required — no new R-ids, no schema-lock amendment). Class: feature-add. Caveman register.

## Ask

ADP today assumes the four root trees (`.aprd/.hld/.adr/.roadmap`) already exist — CLASSIFIER halts with "no spec" if they don't. Projects NOT delivered via ADP (the majority of real brownfield targets) have working code but no ADP foundation: CLASSIFIER blocks, operator has no path forward. Add a first-class intake path that bootstraps the minimal ADP foundation from existing source, then hands off to the normal ADP loop.

## Scope delta (what this CR adds)

### A — ADOPT role (`prompts/00-aprd/ADOPT.md`)

Standalone Phase-0 intake role. Invoked ONCE by operator as a separate explicit run (`adopt` dispatch arg) before the normal feature-add/bugfix run. Never auto-chained.

**Writes:**
- `.aprd/aprd.frozen.md` v1 stub — entities + constraints inferred from code; ACs left OPEN (tagged `[OPEN — client to fill at Checkpoint A]`). NOT full reverse-engineered requirements (P1: request=hypothesis; hallucinating closed ACs is worse than leaving them open).
- `.hld/skeleton.lock` — thin inferred structural lock (component list + seam map, no LLD).
- `.adr/` foundation — minimal ADR stubs for decisions already visible in the codebase (tech stack, key constraints); no speculation.
- Does NOT write `.roadmap/` (RE-RANK bootstraps that from the aprd stub on first normal run).

**Output contract:** minimal bootstrap, not full reverse-engineering. Explicit marker on every OPEN AC. Schema: matches existing `aprd.frozen.md` shape (no new schema id needed; validates against existing `aprd` schema family).

### B — CLASSIFIER delta

Detect `has_adp_artifacts` = false (none of `.aprd/`, `.hld/`, `.adr/` present) → HALT-with-guidance: emit message "No ADP foundation found. Run `adopt` first to bootstrap `.aprd/.hld/.adr/` from existing code, then re-run." Do NOT auto-prepend ADOPT (operator runs two explicit invocations: `adopt` → normal run). Existing greenfield path: `has_adp_artifacts` = true (all three present) → unchanged.

### C — Playbook grounding delta (feature-add + bugfix playbooks)

Both playbooks gain an explicit `has_adp_artifacts` pre-check (true → ADOPT skipped, proceed normally). Existing oracles unaffected — the check is a guard that fires only on the false branch, greenfield fixture trees all have the artifacts present.

### D — Brownfield-adopt oracle (`_fixtures/brownfield-adopt/`)

New fixture subtree:
- `src/` — raw source project (no ADP trees; representative but minimal).
- `expected/` — expected ADOPT bootstrap output (`.aprd/aprd.frozen.md` stub + `.hld/skeleton.lock` + `.adr/` stubs).
- `defects/` — planted-defect variants (closed ACs where stubs required; missing `.hld/skeleton.lock`; ADOPT output clobbering existing artifacts).
- `README.md` — oracle contract + both-directions e2e instructions.

Both-directions e2e: known-good ADOPT run on `src/` produces output matching `expected/` (PASS); planted-defect ADOPT fails value-parity or violates output contract (FAIL). CLASSIFIER on `src/` (no ADP trees) → HALT-with-guidance (PASS); CLASSIFIER on `expected/` (ADP trees bootstrapped) → proceeds normally (PASS).

## Out of scope

- Other unplaybooked CR classes (refactor/migration/perf/investigation still HALT at CLASSIFIER — unaffected).

- Auto-chaining ADOPT into the normal loop (operator override: two explicit runs, not one).
