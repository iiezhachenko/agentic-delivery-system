# CR-011 — controller zero-write guarantee: rule/ADR/CR authoring routes through step-runner

> Operator-directed (2026-06-12), gate Q1–Q3 answered before dispatch. Change against ADP operating model (orchestrator RULES + adp-orchestrator Run section). Class: feature-add. Caveman register.

## Ask

P3 controller-direct carve-out (RM11: orchestrator IS the control loop, not a stochastic-island deliverable) currently does NOT distinguish between (a) mechanical spine ops (git, spine tool shell-outs, lock updates) and (b) stochastic prose authoring (rule text, ADR bodies, CR docs). Result: controller could hand-author decision prose — contradicts zero-write guarantee (RM11: controller picks/dispatches/verifies/gates/promotes, never authors). Operator wants: prose authoring routes through step-runner clean-room, same as any deliverable. Controller-direct writes permitted ONLY for mechanical deterministic ops.

## Gate answers (D20 guarantee 6 — persisted, never re-asked)

- **Q1 Scope:** Lean — orchestrator RULES section (`prompts/_orchestrator.md` line 129 insert) + adp-orchestrator Run section (`.claude/agents/adp-orchestrator.md` line 35 amend) + ADR-0032 (D32 decision) + CR-011.md (this doc).
- **Q2 Trigger:** Immediate, this CR. Discipline tightening. Operator gate rationale: "change must model the discipline it codifies."
- **Q3 Form:** Enumerate permitted controller-direct writes (git ops · spine tool shell-outs · lock re-sign + index update per spine tooling · scratch→destination promotion). Exclude prose authoring class (rule text, ADR bodies, CR docs → step-runner).

## Scope delta (what this CR adds)

### A — Controller zero-write requirement (operating-model requirement)

P3 carve-out (CR-006, W7/W8/W18 precedent) permitted controller-direct mechanical wiring edits (git, spine shell-outs, lock updates). Did NOT explicitly prohibit prose authoring. CR-011 closes gap: controller-direct writes permitted ONLY for mechanical deterministic ops. Rule prose, ADR bodies, CR docs = stochastic authoring class → step-runner clean-room, never hand-authored by controller.

**New R-id (appended to aPRD requirements via this CR doc, same as CR-007's R-MW-*, CR-010's R-LH-*):**
- `R-CW-1` — **Controller zero-write for prose:** controller never hand-authors rule prose, ADR bodies, CR docs. Those route through step-runner clean-room. Permitted controller-direct writes: git ops · spine tool shell-outs (`node tools/<tool>.mjs`) · lock re-sign + index update per spine tooling · scratch→destination promotion. P3 precedent applies to mechanical wiring edits only; does NOT extend to prose authoring of decision rationale.

### B — Orchestrator RULES edit (`prompts/_orchestrator.md`)

New rule inserted after "Controller, not builder" bullet (line 129), before "NEVER run Layer-3 verify inline" bullet. Enumerates R-CW-1 permitted writes + prose exclusion. Cites D32 + P3 scope (mechanical-only, not prose).

### C — adp-orchestrator Run section edit (`.claude/agents/adp-orchestrator.md`)

Run step 4 (line 35) gains R-CW-1 constraint: permitted controller-direct writes enumerated, prose authoring → step-runner.

### D — ADR (controller zero-write decision)

New ADR-0032 (D32): why prose authoring excluded from P3 (zero-write principle — RM11 — controller never authors), scope (permitted mechanical vs excluded stochastic), tradeoffs (allow meta-artifact prose direct — rejected; restrict further — rejected; keep P3 as-is — rejected), consequences (RULES + adp-orchestrator edits, regression-guard).
