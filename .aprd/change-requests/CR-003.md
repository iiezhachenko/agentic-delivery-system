# CR-003 — schema registry fidelity: restore D24 field-docs + tighten constraints

> Operator-directed (2026-06-11), surfaced by the W6-REAUTHOR-REST clean-room gate (03-hld phase). Change request against the **frozen registry** (`schemas/` + `schemas.lock`, the artifact class `ADR-0024`/D24 added). Bumps registry version → re-freeze `schemas.lock` → re-triggers the registry's consumers (the re-authored prompt waves + the validator selftests). Sanctioned immutability exception (workflow §4: change = new version + CR, never hand-edit a frozen body). Caveman register.

## Ask
W1-SCHEMA-REGISTRY externalized every output schema to `schemas/<id>.schema.json` (D24) — but built them **thin**: pure structure, **no field-docs, loose element types**. Violates D24 §1 verbatim: *"Field docs (today inline `//`, AB5) → `description` per property (one machine-readable home)."* Measured across the registry:
- **0 `description` keys** in every `03-hld/*.schema.json` AND `04-build/build-record.schema.json` (only some `00-aprd/*` carry any).
- Loose item shapes: `coverage.seam_realization: {"type":"array"}` (no element schema), `requirements_landed` unconstrained — non-conformant shapes pass the validator.

The inline `## Output schema` block W6 removes carried TWO things the registry was meant to inherit: (a) the `//` **field-docs** (the per-field contract), (b) a concrete **example** that anchored judgment-heavy roles. The registry inherited neither → removing the inline block strands that guidance with no home (AB1 broken: the fact now has ZERO homes, not one).

## Evidence (clean-room, DERIVE-COMPONENTS skeleton pass, greenfield-clean fixture)
- **ORIGINAL prompt (inline schema present), verbatim → 6-box cut ≈ golden** (standalone Persistence/Data-Store C1 + Web Ingress C6; seam map `ingress:C6 · domain:C2 · persistence:C1 · primary_external_integration:C2`). Reproduces the oracle.
- **RE-AUTHORED prompt (schema externalized), verbatim → 5-box cut** (persistence folded onto C2 Identity&Auth) + differently-shaped `requirements_landed`/`seam_realization` — **and the loose validator passed it.**
- Bodies byte-identical → divergence traces to the lost inline-schema guidance, not the role logic. n=1/side (partly stochastic) but the registry gap is objective + supplies the mechanism.

## Root cause
W1-SCHEMA-REGISTRY under-implemented D24 §1. NOT a W6 defect — the re-author is mechanically correct (discriminator clean, schema-ids valid, retained logic byte-identical). The re-author merely **exposes** the latent W1 gap: an inputs-less, inline-schema-less prompt only preserves value-parity if the registry it points to is a FAITHFUL home for what the inline schema carried. Today it is not.

## Scope delta (what the registry version bump adds)
1. **`description` per property** — port every inline `//` field-doc from the pre-CR prompt `## Output schema` blocks into `description` on the matching registry property (D24 §1, one machine-readable home). Source = git history of the prompts (the inline blocks), not re-invention (P5/B11).
2. **Tighten loose constraints** — replace bare `{"type":"array"}` / `{"type":"object"}` on element-bearing fields (`seam_realization`, `requirements_landed`, `entities_owned`, edges, …) with item/property schemas so a non-conformant shape FAILS validation (the divergent re-authored output must not validate-clean). Constraints unfittable in `description` → JSON-Schema construct or `x-*` (D24 §1).
3. **Re-freeze** — new registry version, re-sign `schemas.lock`; old version pinned by its own content_sha256 in history.

## Out of scope
- The decomposition guidance itself (5-vs-6-box): that is the ROLE's `## Rules` job, not the schema's. If post-enrichment the rules still underdetermine the cut, that is a separate prompt-rule finding (route Phase 3), not this CR. This CR only restores the schema to D24 fidelity.
- Re-deciding D24 (Accepted). This CR makes the registry CONFORM to D24, never re-litigates it.
- io-manifest / resolver / det modules (unaffected — read-graph + compute, not output-schema docs).
