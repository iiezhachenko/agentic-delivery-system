# CR-007 — multi-stream workstreams: parallel branches + reconcile + enforce-non-master

> Operator-directed (2026-06-12), gate Q1–Q3 answered before dispatch. Change against ADP operating model (orchestrator + harness wiring). Class: feature-add. Caveman register.

## Ask

ADP today runs all work on master (or one branch). Operator needs:
1. Multiple parallel workstreams, each on its own branch.
2. Each workstream branch auto-merges (reconciles) from main at STEP 0.
3. Orchestrator refuses work on master when a workstream branch is in flight (enforce-non-master).
4. An operator ask UNRELATED to the current workstream spawns a NEW workstream on a NEW branch for a separate harness session to pick up (new-stream dispatch path).

## Gate answers (D20 guarantee 6 — persisted, never re-asked)

- **Q1 Branch:** `feature/multi-stream-workstreams`.
- **Q2 Scope:** FULL CR cascade — aPRD change-request → ADR (branch-isolation decision) → HLD (component/contract delta) → Build (orchestrator delta + new dispatch path).
- **Q3 Reconcile:** BOTH — auto-merge main into workstream branch at STEP 0 AND refuse work on master when a workstream in flight.

## Scope delta (what this CR adds)

### A — Workstream branch model (operating-model requirement)

Each logical unit of work (feature, bugfix, CR wave) runs on a dedicated branch `feature/<slug>` or `bugfix/<slug>`. Master is stable-only (promoted, gate-accepted work). Orchestrator enforces this: if current branch = master AND any workstream branch exists → HALT, name the in-flight branches, instruct operator to check out the right one.

**New R-ids (appended to aPRD requirements):**
- `R-MW-1` — Branch isolation: every workstream runs on a dedicated branch; master = stable landing zone only.
- `R-MW-2` — Auto-reconcile: at STEP 0, orchestrator merges main into the current workstream branch (fast-forward if clean; conflict → HALT, report conflict, do not proceed).
- `R-MW-3` — Enforce-non-master: if current branch = master AND in-flight workstream branches detected → HALT with branch list.
- `R-MW-4` — New-stream dispatch: operator ask unrelated to current workstream → orchestrator writes a `_streams/<slug>/brief.md` stub (captures the ask, branch name, status=pending) and emits a message: "New workstream queued on branch `<branch>`. Check out that branch in a new harness session to continue." Does NOT switch branches in the current session.

### B — Orchestrator delta (`prompts/_orchestrator.md`)

STEP 0 gains two new sub-steps (insert before existing frontier-scan):

**STEP 0.0 — Enforce-non-master (R-MW-3).**
If `git rev-parse --abbrev-ref HEAD` = `master` (or `main`): scan for branches matching `feature/*` or `bugfix/*`. Any present → HALT, list them, instruct operator to check out the target workstream branch.

**STEP 0.1 — Auto-reconcile from main (R-MW-2).**
Run `git fetch origin && git merge origin/main --ff-only`. Clean → continue. Conflict or non-fast-forward → HALT, report conflict details, do not proceed. (No rebase — merge preserves history + is idempotent on re-run.)

**New dispatch path — `new-stream` (R-MW-4).**
Dispatch arg `new-stream <slug> "<brief>"`:
1. Create branch `feature/<slug>` off current HEAD (if not already present).
2. Write `_streams/<slug>/brief.md` with ask, branch, `status: pending`, date.
3. Emit: "New workstream queued on branch `feature/<slug>`. Check out that branch in a new harness session."
4. STOP (do not advance the current workstream).

### C — HLD component/contract delta

New component: **STREAM-MANAGER** (operating-model helper, not a prompt role — implemented as orchestrator sub-steps + `_streams/` tree).

Contracts delta:
- Orchestrator → STEP 0 consumers: precondition gains `branch ≠ master OR no in-flight workstream` (R-MW-3 enforcement point).
- Orchestrator → STEP 0.1: new contract `reconcile_from_main → clean | HALT`.

No new prompt `.md` needed for STREAM-MANAGER (logic lives inline in orchestrator STEP 0 sub-steps; not a stochastic island — mechanical git + file writes). Component entry in `components.json` documents the role; no prompt file (same pattern as Tier-1 emitters: role survives, no stochastic prompt).

### D — ADR (branch-isolation decision)

New ADR-0028: branch-per-workstream model — why dedicated branches over long-lived master-only, how auto-merge + enforce-non-master keep the model coherent, tradeoffs (merge conflicts vs drift, session isolation vs shared history).

## Out of scope

- Merging completed workstream branches back to master (operator action, not automated — same as current promote-to-`prompts/`).
- Conflict resolution automation (HALT + report = correct; resolution is a human decision).
- CI/CD branch policies (out of ADP scope).
- Spine D/A siblings (unchanged).
- Generic orchestrator (`prompts/_orchestrator.generic.md`) — separate loop body; multi-stream applicability deferred (separate CR if needed).
