# CR-005 — orchestrator inline-verify defect: missing hard prohibition on non-spawn verify

> Operator-reported defect (2026-06-12). Class: bugfix. Change against `prompts/_orchestrator.md` (the self-host control loop — loop body only; no frozen artifact version bump required; no new R-ids, no schema-lock amendment).

## Defect

Orchestrator executed verify inline in its own context rather than spawning a fresh `step-runner` subagent. Two failures:

1. **Clean-room isolation broken** — orchestrator holds context from authoring (including golden values it read). Running verify in same context = the "verifier" can unconsciously anchor to known values. Not a true clean-room.
2. **Context bloat** — every layer-3 verify step (resolve, seed, runner execution narration, value-parity check) accumulated in orchestrator context, defeating lean-context design.

## Root cause

`prompts/_orchestrator.md` says "spawn the runner" (STEP 4.1.2) and "spawn a separate verifier subagent" (STEP 4.1.4) but NEVER states:

- a hard prohibition on inline execution in the orchestrator's own context, and
- that both runner AND verifier must be separate `step-runner` spawns.

The prohibition is implied (clean-room = isolated context) but not a literal rule. An orchestrator following only the letter could narrate verify work inline and satisfy every explicit instruction. The implicit becomes a gap at runtime.

SUBAGENT CONTRACT (the "every subagent is step-runner" paragraph) reinforces the intent but still does not forbid inline execution.

## Fix scope

One targeted edit to `prompts/_orchestrator.md` — 3 homes (AB1-compliant: all directive/mechanical, not re-stated reasoning):

1. **STEP 4.1.2** — spawn-runner instruction gains inline `(MUST be a separate spawn — not inline)` prohibition literal.
2. **STEP 4.1.4** — verifier spawn instruction restated `Verifier MUST be a different spawn than the runner`.
3. **RULES block** — hard rule added: `NEVER run Layer-3 verify inline. Runner + verifier MUST each be a step-runner subagent spawn. Inline execution = broken clean-room + context bloat. No exception.`

3 homes because STEP 4.1.2 covers the runner procedure, STEP 4.1.4 covers the verifier procedure, and RULES is the unconditional guard — each slot is distinct; no fact restated in the same slot.

## Out of scope

- `adp-orchestrator.md` agent def — already correctly says "spawns step-runner subagents"; no change.
- `step-runner.md` agent def — unchanged.
- Any frozen artifact, schema, or lock — unchanged.
- The generic orchestrator (`prompts/_orchestrator.generic.md`) — separate loop body; audit deferred (no report of the same defect there).
