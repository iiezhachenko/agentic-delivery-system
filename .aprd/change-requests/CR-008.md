# CR-008 — workstream branch binding: explicit association + wrong-branch auto create/checkout

> Operator-directed (2026-06-12). Refinement of CR-007 multi-stream domain. Class: feature-add. Caveman register.

## Ask

CR-007 bound each workstream to a branch via `_streams/<slug>/brief.md` (fields: ask, branch, status, date). But orchestrator STEP 0 does NOT read that record at run-time to verify it's on the expected branch. "Wrong branch" is currently undefined — STEP 0.0 only catches master-with-in-flight. When operator is on a workstream branch but the wrong one (e.g., feature/A while `_streams/B/brief.md` says expected branch = feature/B), nothing fires.

Request:
1. Each workstream explicitly associated with a specific branch (brief.md already captures `branch:` field — needs to be the authoritative binding, not just documentation).
2. When repo is on the wrong branch for the active workstream, auto create-or-checkout the correct branch (not just HALT).

Reconcile with D28 enforce-non-master safety invariant: the "wrong branch" fix MUST NOT silently undermine the master-HALT guard. Brief record names the expected branch; mismatch → checkout that branch if it exists, else create it. Bare-master-with-in-flight still HALTs unless a target stream is unambiguous.

## Design options (gate question — operator must choose Q2)

Three options, all preserve D28 enforce-non-master invariant.

### Option A — Session dispatch with slug arg (explicit, requires operator input each run)

Orchestrator gains new dispatch arg `run <slug>` (or existing dispatch arg extended). At STEP 0, reads `_streams/<slug>/brief.md`, confirms `branch:` field, runs `git checkout -B feature/<slug>` (create if absent). No ambiguity — operator names the stream. STEP 0.0 unchanged (still HALTs on master-with-in-flight when no slug given).

- Pro: unambiguous, composable, no inference.
- Con: operator must always specify slug — breaks current zero-arg default-advance behavior.

### Option B — STEP 0.0 unambiguous auto-checkout (recommended; most ergonomic)

Extend STEP 0.0: after `git rev-parse --abbrev-ref HEAD` check:
1. If on correct branch (matches any `_streams/*/brief.md` `branch:` field) → proceed (pass, no action).
2. If on master AND exactly ONE `_streams/*/brief.md` with `status: pending` → auto-checkout that stream's `branch:` (create if absent), continue.
3. If on master AND zero pending streams → proceed (solo-master repo, STEP 0.0 pass, current behavior preserved).
4. If on master AND >1 pending streams → HALT as today (ambiguous; list them; instruct operator to check out target).
5. If on wrong non-master branch (branch present, but `_streams/*/brief.md` `branch:` points elsewhere) → auto-checkout the correct one from the matching brief.

- Pro: common case (one pending stream, on master or mismatched) automatic. No operator action needed.
- Con: adds inference; case 5 requires reading ALL `_streams/` briefs to find a match — must handle zero-match (no matching brief → HALT with "branch not registered in _streams; run new-stream first").

### Option C — Explicit `switch-to <slug>` dispatch path (additive, safest)

New dispatch arg `switch-to <slug>`: reads `_streams/<slug>/brief.md`, checks out or creates `branch:` field value. Emits confirmation. STOP (operator re-runs on correct branch). STEP 0.0 completely unchanged. No inference at STEP 0.

- Pro: purely additive; zero regression surface; composes with B later.
- Con: requires two operator invocations (switch-to + re-run) vs one; less ergonomic.

### Recommended default if operator has no preference

**Option B** — highest ergonomic value, reconciles cleanly with D28 (cases 3+4 preserve current behavior, only cases 2+5 add auto-checkout). The STEP 0.0 change is well-bounded and oracle-testable.

## Scope delta (pending Q2 answer — Option B framing)

### A — brief.md `branch:` field as authoritative binding

`brief.md` `branch:` is already written by `new-stream` dispatch. CR-008 elevates it from documentation to authoritative binding: STEP 0.0 reads it. No schema change needed — field already exists.

### B — STEP 0.0 delta (Option B)

STEP 0.0 gains branch-match logic:
1. Read all `_streams/*/brief.md` where `status: pending` → collect `branch:` values.
2. Current HEAD branch in that set → correct, proceed.
3. HEAD = master/main:
   - 0 pending → proceed (solo-master flow unchanged).
   - 1 pending → auto-checkout its `branch:` (create if absent), continue.
   - >1 pending → HALT (ambiguous; list).
4. HEAD not in set AND not master → HALT: "branch `<HEAD>` not registered; run new-stream or switch branches manually."

New R-id: **R-MW-5** — wrong-branch auto-checkout: if current branch does not match any registered pending stream AND exactly one pending stream exists OR HEAD = master with one pending → auto create/checkout the correct branch.

### C — new `switch-to` dispatch path (Option C, additive regardless of Q2)

`switch-to <slug>`: validates `_streams/<slug>/brief.md` exists → checkout/create `branch:` field → emit "Switched to branch `<branch>`. Re-run orchestrator to continue workstream `<slug>`." STOP. No state change to current workstream. Safe to ship alongside Option B.

### D — HLD delta

`STREAM-MANAGER` sentinel in `components.json` + `contracts.json` `step-0-precondition` contract updated to reflect branch-match logic.

### E — Oracle delta

`_fixtures/multi-stream/` gains new scenarios:
- `wrong-branch-auto-checkout/` — HEAD on wrong feature branch → auto-checkout correct one, proceed.
- `master-single-pending-auto-checkout/` — master + one pending stream → auto-checkout, proceed.
- `master-multi-pending-halt/` — master + >1 pending → HALT (validates ambiguity guard).

New defect:
- `no-wrong-branch-checkout/` — auto-checkout logic removed → HALT instead of checkout.

### F — ADR amendment

ADR-0028 gains Amendment-1: branch-binding policy (R-MW-5). Documents the brief.md→STEP 0.0 authoritative binding + auto-checkout decision + the tradeoffs (inference vs ergonomics, ambiguity guard).
