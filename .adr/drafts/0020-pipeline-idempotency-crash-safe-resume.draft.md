## Decision

- **D20 — Pipeline idempotency & crash-safe resume (RESOLVED 2026-06-08).** System-level architecture decision (not an authoring call): the pipeline MUST resume cleanly from ANY interruption — lost connection, killed agent, crash — **mid-step**, not only at clean slice/phase boundaries. Idempotency is a binding obligation on the BUILD-phase prompts (IMPLEMENT, INTEGRATE, MATERIALIZE-ORACLE, VERIFY-OUTPUT) + every artifact-writing role + the orchestrator, derived from the disk-as-source-of-truth principle (`docs/self-host-workflow.md` §10; D3 artifacts-on-disk). Surfaced by the end-user resilience requirement (`docs/generic-usage-guide.md` §3). **Guarantees (all binding):** **(1) Disk is the SOLE source of truth** — no run depends on in-memory agent state; progress = the on-disk artifact tree only. **(2) Atomic writes** — every role writes its output to a temp path then atomically renames/swaps into the declared `outputs` path. A mid-write interruption never leaves a partial/corrupt artifact at the canonical path: the file is either the complete, schema-valid result or absent. (Closes the gap that resumable auto-select alone — D14/D16/D17/D18/D19 sentinels = "output present" — does NOT cover: a half-written present-but-corrupt file.) **(3) Frozen artifacts immutable; steps only ADD** — locks (`aprd.lock`/`adr.lock`/`skeleton.lock`/…) + accepted-slice/build records are never mutated; a role only adds new artifacts, so an interrupted or re-run step cannot damage committed/agreed/accepted state (makes the existing freeze discipline an explicit invariant). **(4) Step re-run = idempotent** — recovery unit is ONE step (one role); a re-run reads only frozen inputs + (over)writes its single output via atomic swap → equivalent result, last-writer-wins, no double-count. Benign run-to-run variance still allowed (correctness = schema + ID-thread + invariants, NOT byte-equality — the standing variance rule). **(5) Resume = re-derive the frontier from disk** — orchestrator scans the tree, schema-VALIDATES the frontier (partial/invalid ⇒ "not done"), and resumes at the FIRST step whose declared output is absent OR invalid; valid completed steps skipped. Generalizes the per-increment resumable auto-select (D14) + IMPLEMENT build-record resume (D12) from clean boundaries to mid-step crashes, and ADDS the validity check (presence alone is insufficient — see (2)). **(6) Gate replies persisted on receipt** — interactive gate inputs (`06-answers.md`, `07-sequence-reviewed.json`, demo-acceptance record) written to disk the moment received; resume reads a persisted reply (no re-ask), interrupted-before-persist ⇒ re-ask (safe — never a silent guess). Binds the QUESTION-GEN gate / SEQUENCE-REVIEW / DEMO-GEN. **(7) External side effects idempotent** — a slice's staging deploy/integration MUST be safe to re-apply (re-deploying the same build reproduces the same staging state); the build-record + commit is the slice commit point. **Consequences:** these become load-bearing Rules in the BUILD-phase prompts (+ orchestrator); the windowed/e2e test protocol gains an interrupt-mid-step → resume scenario (verify (2)/(5): kill after a partial write, confirm resume re-runs the step to a clean artifact). Reopen if a step proves genuinely non-idempotent (e.g. an unavoidable non-idempotent external API) — that step then needs an explicit dedupe/idempotency-key recorded here.
